# ADR-005: Resend Verification No Longer Regenerates Password

## Status

Accepted

## Date

2026-06-12

## Context

The current `EmailVerificationController@resend` method regenerates the user's password (random 12-char string) on every resend request. This was likely intended as a security measure, but it creates a poor UX — users who already know their password get it silently changed, and the email they receive contains a new password that differs from the original one sent at registration.

## Decision

Remove password regeneration from the resend verification flow. The resend endpoint will:

1. Generate a new verification token
2. Send an email containing ONLY the verification link
3. NOT modify the user's password
4. Keep reCAPTCHA v3 protection and rate limiting

Users who forgot their password should use the existing "Forgot password" flow (`PasswordResetController`). The resend endpoint's sole purpose is to deliver a fresh verification link.

## Alternatives Considered

### Keep regeneration but warn user

- **Description**: Regenerate password and send it in the email with a warning
- **Pros**: User gets both verification link and a working password in one email
- **Cons**: Still changes password without user request; confusing if user already memorized their password
- **Why rejected**: Violates principle of least surprise; the "Forgot password" flow exists for this purpose

### Hybrid: ask user at time of resend

- **Description**: Frontend asks "Did you forget your password too?" before resending
- **Pros**: User chooses the behavior
- **Cons**: Adds UI complexity; user may not know if they want a new password
- **Why rejected**: Over-engineered; keep each endpoint single-purpose

## Consequences

- WelcomeMail email template must be updated to not include password in resend scenario
- The initial registration email still includes the generated password (that stays)
- Users who lose their password before verifying will need to use "Forgot password" after verification, or the system can send a password reset link proactively after verification
- Simpler, more predictable UX
