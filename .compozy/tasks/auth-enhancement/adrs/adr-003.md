# ADR-003: OTP and Magic Link Storage Strategy

## Status

Accepted

## Date

2026-06-12

## Context

The PRD specifies OTP (6-digit code) and magic link (signed URL) for passwordless login. We need to decide whether to store tokens in the database or encode them as self-contained signed tokens (JWT).

## Decision

Store both OTP codes and magic link tokens in the `otp_tokens` table with hashed values.

- OTP: 6-digit numeric code, hashed with bcrypt before storage
- Magic link: random 64-character token, hashed before storage
- Both stored in the same `otp_tokens` table with a `type` discriminator
- Tokens are single-use (marked via `used_at` timestamp)
- Cleanup: a scheduled artisan command removes expired records daily

## Alternatives Considered

### Self-contained signed tokens (JWT)

- **Description**: Both OTP and magic link encoded as signed JWTs containing email and expiry
- **Pros**: No DB writes, no cleanup needed
- **Cons**: Cannot manually revoke tokens; cannot rate-limit OTP attempts without DB; OTP comparison harder without storage
- **Why rejected**: Rate limiting and revocation are requirements; DB storage gives more control

### Magic link as signed token, OTP in DB

- **Description**: Hybrid approach — magic link uses self-contained token, OTP stored in DB
- **Pros**: Magic link validation is stateless
- **Cons**: Two different validation mechanisms adds complexity; inconsistency
- **Why rejected**: Simpler to have one uniform approach; both benefit from DB-level rate limiting

## Consequences

- DB cleanup mechanism needed (cron for expired tokens)
- Rate limiting on OTP attempts is straightforward (count failed attempts per email from DB)
- Tokens can be revoked manually if needed
- Bcrypt hashing of 6-digit codes is acceptable (not performance-critical at this scale)
