# ADR-004: Google New User Onboarding Flow

## Status

Accepted

## Date

2026-06-12

## Context

When a user authenticates via Google OAuth but no account exists with that email, the system needs to decide how to handle first-time registration. The PRD specifies a post-Google onboarding step.

## Decision

Implement a post-Google authentication onboarding flow:

1. User clicks "Sign in with Google" and authorizes
2. Backend receives Google user data, detects no existing user
3. Backend creates a temporary session (signed token with Google data, valid 15 minutes)
4. Frontend redirects to `/admin/onboarding` with the temporary token
5. Onboarding form collects: store name, store slug, WhatsApp number
6. User submits form; backend creates Tenant and User with email_verified_at = now, google_id linked
7. Backend issues final Sanctum token; user is redirected to dashboard

## Alternatives Considered

### Only for existing users

- **Description**: Google login only works if email already registered
- **Pros**: No onboarding form needed; simpler
- **Cons**: Excludes new users who want a quick sign-up; defeats purpose of "login with Google"
- **Why rejected**: Too restrictive; onboarding with Google is a common UX pattern

### Collect data from Google scopes

- **Description**: Use Google profile data (name) to auto-create account, generate slug from name
- **Pros**: No onboarding needed
- **Cons**: Cannot reliably generate store slug (may conflict, user may want custom); WhatsApp number not available from Google
- **Why rejected**: Missing critical data (WhatsApp) and custom slug preference

## Consequences

- Temporary token mechanism needed for secure onboarding state
- Frontend onboarding page reuses existing registration form fields
- Google-first users will have one extra step after authorization
- Tenant creation logic can be partially reused from RegistrationController
