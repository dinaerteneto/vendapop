# ADR-001: Progressive Expansion Authentication Approach

## Status

Accepted

## Date

2026-06-12

## Context

The VesteZap platform currently supports only email+password login with email verification. Users who register but do not receive the verification email have no way to resend it from the login screen. Additionally, the platform lacks alternative login methods such as Google OAuth and email-based OTP/magic link, which are market expectations for modern SaaS platforms.

The user wants to solve three problems:
1. Allow resending verification email directly from the login screen when a 403 error occurs
2. Add Google OAuth as an alternative login method
3. Add OTP + magic link login via email

## Decision

Adopt **Approach A — Progressive Expansion**: evolve the existing login form by progressively adding the three features as complementary elements, keeping the core email+password flow intact.

- **Resend**: inline button appears on the login form when the 403 error is returned, protected by reCAPTCHA v3 and rate limiting
- **Google**: "Sign in with Google" button placed below the password form; when the Google email matches an existing unverified account, prompt the user to link
- **OTP + Magic Link**: "Send code by email" link on the login screen; user enters email, receives a 6-digit code + magic link, and logs in without a password

## Alternatives Considered

### Alternative B: Unified Tab Screen

- **Description**: Three tabs on the same screen — "Password", "Google", "Code by Email"
- **Pros**: All methods visible at once, clear separation
- **Cons**: More visual complexity for users who only want password login; tab switching adds friction
- **Why rejected**: Over-engineered for the current user base; the progressive approach is simpler and keeps the main flow intact

### Alternative C: Login-step Progression

- **Description**: Two-step flow — first enter email, then choose method
- **Pros**: Guided UX, good for mobile
- **Cons**: Extra clicks for returning users; more complex frontend state management
- **Why rejected**: Adds unnecessary friction for the primary use case (password login); progressive expansion is more efficient

## Consequences

### Positive

- Minimal disruption to existing users — the password login flow remains identical
- Each feature can be developed and shipped independently
- New login methods are discoverable but not intrusive
- Reuses existing reCAPTCHA v3 integration already present in registration

### Negative

- Login page will grow in vertical space as features are added
- Google OAuth and OTP flows need their own UI states and error handling

### Risks

- **Rate limiting abuse**: mitigated by reCAPTCHA v3 + server-side rate limiting on resend and OTP endpoints
- **Google OAuth linking confusion**: mitigated by a clear prompt with "Yes, link" / "No, use another email" options
- **OTP delivery failure**: mitigated by showing the magic link as fallback in the same email

## Implementation Notes

- Backend: Laravel Socialite for Google OAuth; custom OTP generation (6-digit code + signed magic link URL)
- Frontend: Extend SignIn.tsx with conditional UI states; Google Sign-In button via Socialite redirect or Google Identity Services library
- Database: Add `google_id`, `otp_code`, `otp_expires_at` columns to users table (or a separate otp_tokens table)
- Rate limit: 1 resend per 2 minutes per email, tracked in cache/DB
