# ADR-002: Database Schema for Google OAuth and OTP Storage

## Status

Accepted

## Date

2026-06-12

## Context

The authentication enhancement requires storing Google OAuth credentials and OTP/magic link tokens. The existing `users` table has basic fields (tenant_id, name, email, password, is_owner, email_verified_at). We need to decide where to store Google OAuth data and OTP tokens.

## Decision

- Add `google_id`, `google_token`, `google_refresh_token` columns directly to the `users` table
- Create a new `otp_tokens` table for OTP codes and magic link tokens

### `users` table additions

| Column | Type | Nullable | Description |
|--------|------|----------|-------------|
| google_id | string (VARCHAR) | yes | Google's user ID |
| google_token | text (TEXT) | yes | Google OAuth access token |
| google_refresh_token | text (TEXT) | yes | Google OAuth refresh token |

### `otp_tokens` table

| Column | Type | Description |
|--------|------|-------------|
| id | bigint (PK) | Auto-increment |
| email | string (VARCHAR) | User's email |
| code | string (VARCHAR) | Hashed 6-digit OTP code |
| magic_link_token | string (VARCHAR) | Hashed magic link token |
| type | string (VARCHAR) | 'otp' or 'magic_link' or 'both' |
| expires_at | datetime | Token expiration |
| used_at | datetime (nullable) | When token was consumed |
| created_at | datetime | When token was created |

## Alternatives Considered

### Separate user_social_logins table

- **Description**: Normalized table for all social logins (provider, provider_id, tokens)
- **Pros**: Fully normalized, supports multiple providers easily
- **Cons**: Extra JOIN on every Google login, more complex queries
- **Why rejected**: Only one OAuth provider planned; YAGNI — keep it simple with columns on users table

## Consequences

- Google auth check is a simple column lookup on users table
- Adding future OAuth providers would require either new columns or the normalized approach
- OTP tokens table will need periodic cleanup of expired records (cron job)
- Migration required for existing users table
